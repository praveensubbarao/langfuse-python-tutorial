The complete data model
trace → span → generation — how they nest

In Hour 1 you saw a trace with one generation inside it. That covered the simplest case. Real apps have much deeper hierarchies. Here's the full picture:

TRACE ← the entire user request lifecycle. Defined by its root span. │ user_id "user-123" │ session_id "session-abc" │ tags ["qa", "architect"] │ ├─ SPAN ← a unit of non-LLM work (retrieval, pre-processing, API call) │ │ name "retrieve-context" │ │ input {query: "..."} │ │ output {chunks: [...]} │ │ latency 45ms │ │ │ └─ GENERATION ← specifically an LLM call. Child of the span above. │ model "llama3.2" │ input {messages: [...]} │ output "The answer is..." │ usage.input 312 tokens │ usage.output 148 tokens │ model_parameters {temperature: 0.7} │ latency 1840ms │ └─ SPAN ← another sibling step (post-processing, formatting, saving) name "format-response" latency 12ms

The key rule: nesting is determined by which observation is "active" when you create a new one. The currently active span becomes the parent of any new span or generation created inside it. OpenTelemetry context propagation handles this automatically.

Type When to use Unique fields
TRACE
defined by root span		 Wraps the entire operation. Created implicitly by the first span you open. user_id, session_id, tags via propagate_attributes()
SPAN Any non-LLM unit of work: DB lookups, vector retrieval, API calls, pre/post-processing steps. input, output, metadata, level (DEBUG/INFO/WARNING/ERROR)
GENERATION Every LLM call. Langfuse treats it specially for token tracking, cost, and evals. model, model_parameters, usage, prompt_name, completion_start_time
v4 architecture note: Langfuse Python SDK v4 is built on OpenTelemetry. A "Langfuse trace" maps to an "OTel trace", a "Langfuse span" maps to an "OTel span with Langfuse-specific attributes", and a "generation" is just a span with gen_ai.* attributes attached. This means any other OTEL-instrumented library (HTTP clients, DBs) can appear inside your Langfuse traces automatically.

Method 1: context managers
start_as_current_observation() — explicit, full control

The primary v4 way to create observations is start_as_current_observation() as a context manager. The with statement guarantees the span is properly closed even if an exception occurs, and everything created inside is automatically a child.

Create 03_context_managers.py:

PYTHON 03_context_managers.py 
from dotenv import load_dotenv
import os
load_dotenv()

from langfuse import get_client, propagate_attributes
from langfuse.openai import OpenAI

langfuse = get_client()

client = OpenAI(
    base_url=os.getenv("OLLAMA_BASE_URL"),
    api_key="ollama",
)

# ─────────────────────────────────────────────────────────────
# FULL TRACE: root span → retrieval span → generation (nested)
# ─────────────────────────────────────────────────────────────
with propagate_attributes(
    trace_name="qa-pipeline",
    user_id="user-123",
    tags=["qa", "hour-2"],
    metadata={"tutorial": "day1-hour2"},
):
    # Root span — this DEFINES the trace
    with langfuse.start_as_current_observation(
        as_type="span",
        name="handle-user-question",
        input={"question": "What is boundary value analysis?"},
    ) as root_span:

        # ── Step 1: Simulate document retrieval ──────────────────
        with langfuse.start_as_current_observation(
            as_type="span",
            name="retrieve-context",
            input={"query": "boundary value analysis"},
        ) as retrieval_span:

            # Simulate finding relevant docs (no real DB needed)
            fake_docs = [
                "BVA tests at the edges of input domains...",
                "Equivalence partitioning divides inputs into classes...",
            ]
            retrieval_span.update(
                output={"chunks_found": 2, "docs": fake_docs},
                metadata={"source": "knowledge-base"},
            )

        # ── Step 2: LLM generation (child of root, sibling of retrieval) ─
        with langfuse.start_as_current_observation(
            as_type="generation",
            name="generate-answer",
            model=os.getenv("OLLAMA_MODEL"),
            model_parameters={"temperature": 0.3},
            input={
                "system": "You are a QA expert. Use the context provided.",
                "context": fake_docs,
                "user": "What is boundary value analysis?",
            },
        ) as gen:

            response = client.chat.completions.create(
                model=os.getenv("OLLAMA_MODEL"),
                messages=[
                    {"role": "system", "content": "You are a QA expert."},
                    {"role": "user",   "content": "What is boundary value analysis?"},
                ],
                temperature=0.3,
            )
            answer = response.choices[0].message.content

            # Update generation with the output after the call
            gen.update(output=answer)

        # ── Step 3: Set final trace output ───────────────────────────
        root_span.update(output={"answer": answer})

print("Answer:", answer)
langfuse.flush()
BASH run it 
$ python 03_context_managers.py

In your Langfuse dashboard, click the new trace. You'll see the timeline view show three nested observations: handle-user-question at the top, with retrieve-context and generate-answer nested inside it as children. The latency bars show exactly how long each step took.

Why is as_type important? Setting as_type="generation" tells Langfuse this span is an LLM call. It unlocks the token usage panel, cost estimation, and makes it eligible for LLM-as-a-judge evaluators in Day 2. Without it, your LLM call is just a plain span.

Method 2: the @observe decorator
cleaner functions, same power

The @observe decorator is sugar on top of context managers. It automatically wraps a function in a span, captures its arguments as input and its return value as output, and properly nests with any parent observation that's active when the function is called.

CONTEXT MANAGER
Explicit. You control exactly what input/output gets logged. Best for complex logic, custom error handling, or when you need the span object mid-function.
@OBSERVE DECORATOR
Clean. Auto-captures function args and return value. Best for simple functions where the args/return are exactly what you want to log.

Create 04_observe_decorator.py:

PYTHON 04_observe_decorator.py 
from dotenv import load_dotenv
import os
load_dotenv()

from langfuse import observe, get_client, propagate_attributes
from langfuse.openai import OpenAI

langfuse = get_client()
client = OpenAI(base_url=os.getenv("OLLAMA_BASE_URL"), api_key="ollama")


@observe(name="retrieve-docs")  # creates a SPAN automatically
def retrieve_context(query: str) -> list[str]:
    # Function args become the span input, return value becomes output
    # Simulate retrieval (replace with real vector DB in your app)
    return [
        f"Test technique relevant to: {query}",
        "Edge cases should always be tested at boundaries.",
    ]


@observe(name="llm-generation", as_type="generation")
def generate_answer(question: str, context: list[str]) -> str:
    system = "You are a QA Architect. Use the context to answer concisely."
    user_msg = f"Context:\n{chr(10).join(context)}\n\nQuestion: {question}"

    # Update the generation with structured input for better UI display
    langfuse.update_current_generation(
        model=os.getenv("OLLAMA_MODEL"),
        model_parameters={"temperature": 0.5},
        input={"system": system, "user": user_msg},
    )

    response = client.chat.completions.create(
        model=os.getenv("OLLAMA_MODEL"),
        messages=[
            {"role": "system", "content": system},
            {"role": "user",   "content": user_msg},
        ],
        temperature=0.5,
    )
    return response.choices[0].message.content


@observe(name="qa-pipeline")  # top-level trace
def run_qa_pipeline(question: str) -> str:
    # Each call below automatically becomes a CHILD of this function's span
    context = retrieve_context(question)
    answer  = generate_answer(question, context)
    return answer


# propagate_attributes wraps the TOP-LEVEL call (the correct v4 pattern)
if __name__ == "__main__":
    with propagate_attributes(
        trace_name="QA Architect Query",
        user_id="user-123",
        tags=["qa", "decorator-demo"],
        metadata={"tutorial": "day1-hour2"},
    ):
        result = run_qa_pipeline("Explain equivalence partitioning.")

    print(result)
    langfuse.flush()
BASH run it 
$ python 04_observe_decorator.py

In the dashboard, click the new trace. You'll see the clean tree: run_qa_pipeline at top (the trace root), with retrieve-docs and llm-generation as children — their inputs are your function arguments, outputs are return values. No extra code needed.

Performance note: @observe captures all function arguments by default. If you pass large objects (big lists, DataFrames, raw documents), this can add overhead. Disable auto-capture per function: @observe(capture_input=False, capture_output=False), then manually set input/output with update_current_generation().

Nesting spans correctly
the one rule that makes everything work

Nesting works by OpenTelemetry context propagation. Whichever span is currently "active" in the context becomes the parent of anything new you create inside it. Here's what correct vs incorrect nesting looks like:

PYTHON nesting_rules.py — read, don't run 
# ✅ CORRECT: nested with-blocks create a parent-child tree
with langfuse.start_as_current_observation(as_type="span", name="parent") as parent:
    with langfuse.start_as_current_observation(as_type="span", name="child-a"):
        pass   # child-a is nested under parent ✓
    with langfuse.start_as_current_observation(as_type="generation", name="child-b"):
        pass   # child-b is sibling of child-a, both under parent ✓

# Resulting tree:
#   parent (SPAN)
#   ├── child-a (SPAN)
#   └── child-b (GENERATION)


# ✅ CORRECT: @observe automatically nests based on call stack
@observe(name="inner")
def inner_fn(): ...

@observe(name="outer")
def outer_fn():
    inner_fn()   # inner_fn span is a child of outer_fn span ✓


# ❌ WRONG: mixing decorator and propagate_attributes inside the function
@observe(name="outer")
def bad_fn():
    with propagate_attributes(user_id="x", tags=["y"]):
        return inner_fn()   # tags WON'T appear in UI — see Hour 1 fix
# ✅ Fix: put propagate_attributes OUTSIDE the @observe call
with propagate_attributes(user_id="x", tags=["y"]):
    bad_fn()   # now the attributes propagate to all child spans ✓


# ❌ WRONG: sequential blocks are SIBLINGS not children
with langfuse.start_as_current_observation(as_type="span", name="step-1"): pass
with langfuse.start_as_current_observation(as_type="span", name="step-2"): pass
# step-1 and step-2 are SIBLINGS (separate root spans = separate traces!)
# To make them children of the same trace, nest them inside a root span.
The golden rule: If you want two spans in the same trace, they must both be created while a common root span is active. Sequential with blocks at the top level create separate traces — always wrap them in a parent.

Enriching generations
model_parameters · usage · levels · metadata

Generations have richer fields than regular spans. The more you fill in, the more useful Langfuse becomes — especially for cost tracking and eval workflows. Create 05_rich_generation.py:

PYTHON 05_rich_generation.py 
from dotenv import load_dotenv
import os, time
load_dotenv()

from langfuse import get_client, propagate_attributes
from langfuse.openai import OpenAI
from datetime import datetime, timezone

langfuse = get_client()
client  = OpenAI(base_url=os.getenv("OLLAMA_BASE_URL"), api_key="ollama")

SYSTEM_PROMPT = (
    "You are an experienced QA Architect. "
    "Answer in structured bullet points."
)

with propagate_attributes(
    trace_name="rich-generation-demo",
    user_id="user-123",
    tags=["qa", "rich-demo"],
    metadata={"tutorial": "day1-hour2-step5"},
):
    with langfuse.start_as_current_observation(
        as_type="span",
        name="qa-request",
        input={"question": "List 5 types of software testing."},
    ) as root:

        # Track when the request reaches the LLM (for streaming latency)
        request_start = datetime.now(timezone.utc)

        with langfuse.start_as_current_observation(
            as_type="generation",
            name="list-testing-types",
            model=os.getenv("OLLAMA_MODEL"),

            # Model parameters — shown in Langfuse generation panel
            model_parameters={
                "temperature": 0.2,
                "max_tokens": 400,
            },

            # Structured input — shows nicely in the UI
            input=[
                {"role": "system", "content": SYSTEM_PROMPT},
                {"role": "user",   "content": "List 5 types of software testing."},
            ],

            # Custom metadata for this generation
            metadata={
                "prompt_version": "1.0",
                "use_case": "qa-education",
            },
        ) as gen:

            response = client.chat.completions.create(
                model=os.getenv("OLLAMA_MODEL"),
                messages=[
                    {"role": "system", "content": SYSTEM_PROMPT},
                    {"role": "user",   "content": "List 5 types of software testing."},
                ],
                temperature=0.2,
                max_tokens=400,
            )
            answer = response.choices[0].message.content

            # Update generation with output + token usage
            gen.update(
                output=answer,
                # Manually log token counts (Langfuse uses this for cost tracking)
                usage={
                    "input":  response.usage.prompt_tokens,
                    "output": response.usage.completion_tokens,
                    "total":  response.usage.total_tokens,
                },
            )

        # Set the root span output
        root.update(output={"answer": answer})

print(answer)
langfuse.flush()
BASH run it 
$ python 05_rich_generation.py

In the dashboard, click this trace's generation. You'll now see the Usage panel with token counts, the Model Parameters panel showing temperature and max_tokens, and Metadata with your custom fields. This is the foundation for cost tracking across thousands of requests.

Token usage and Ollama: Ollama does return usage in its responses (same field names as OpenAI). The code above reads response.usage.prompt_tokens and response.usage.completion_tokens. If you ever see None, add options={"num_predict": 400} to the Ollama request — some models need this hint to return usage stats.

Error handling & span levels
capture failures as ERROR spans — not silent gaps

One of Langfuse's biggest practical values: when your LLM pipeline fails, you see exactly which step failed and with what error. Context managers make this automatic — the span closes properly even if an exception is thrown. You just need to mark it as an error.

PYTHON 06_error_handling.py 
from dotenv import load_dotenv
import os
load_dotenv()

from langfuse import get_client, propagate_attributes
from langfuse.openai import OpenAI

langfuse = get_client()
client  = OpenAI(base_url=os.getenv("OLLAMA_BASE_URL"), api_key="ollama")


def risky_retrieval(query: str) -> list[str]:
    # Simulate a retrieval that might fail
    if "crash" in query.lower():
        raise ValueError(f"Retrieval service unavailable for query: {query}")
    return ["Relevant test case documentation..."]


def run_with_error_handling(question: str):
    with propagate_attributes(
        trace_name="error-demo",
        user_id="user-123",
        tags=["error-handling"],
        metadata={"tutorial": "day1-hour2-step6"},
    ):
        with langfuse.start_as_current_observation(
            as_type="span",
            name="pipeline-with-error",
            input={"question": question},
        ) as root:

            try:
                with langfuse.start_as_current_observation(
                    as_type="span",
                    name="retrieval-step",
                    input={"query": question},
                ) as retrieval_span:

                    try:
                        docs = risky_retrieval(question)
                        retrieval_span.update(
                            output={"docs": docs},
                            level="DEFAULT",   # DEFAULT | DEBUG | WARNING | ERROR
                        )
                    except ValueError as e:
                        # Mark the retrieval span as ERROR — visible in dashboard
                        retrieval_span.update(
                            level="ERROR",
                            status_message=str(e),
                            output={"error": str(e)},
                        )
                        # Graceful degradation: continue without context
                        docs = []

                # Continue to LLM even if retrieval failed
                with langfuse.start_as_current_observation(
                    as_type="generation",
                    name="llm-fallback",
                    model=os.getenv("OLLAMA_MODEL"),
                    input={"context_available": bool(docs), "question": question},
                ) as gen:

                    fallback_msg = (
                        "Answer from knowledge only (retrieval failed): "
                        if not docs else ""
                    )
                    response = client.chat.completions.create(
                        model=os.getenv("OLLAMA_MODEL"),
                        messages=[{
                            "role": "user",
                            "content": f"{fallback_msg}{question}",
                        }],
                    )
                    answer = response.choices[0].message.content
                    gen.update(output=answer)

                root.update(output={"answer": answer, "retrieval_failed": not bool(docs)})
                return answer

            except Exception as e:
                # Catastrophic failure — mark the root span as error
                root.update(level="ERROR", status_message=str(e))
                raise


if __name__ == "__main__":
    # Test 1: normal query
    result = run_with_error_handling("What is smoke testing?")
    print("Normal query:", result[:100])

    # Test 2: query that triggers retrieval failure
    result = run_with_error_handling("crash: What is regression testing?")
    print("Degraded query:", result[:100])

    langfuse.flush()
BASH run it 
$ python 06_error_handling.py

In the dashboard you'll now see two traces. The second one shows the retrieval-step span highlighted in red with the error message, while the llm-fallback generation still succeeded. This is exactly what production debugging looks like — the trace tells you the LLM worked fine, the retrieval didn't.

Hour 2 mini-project: QA Architect pipeline
putting it all together in one file

Combine everything from this hour into a realistic QA Architect assistant that has: a root trace with user metadata, a retrieval span, a generation with rich fields, and error handling. This is the pattern you'll extend in Hours 3 and beyond.

PYTHON 07_qa_architect_pipeline.py 
from dotenv import load_dotenv
import os
load_dotenv()

from langfuse import observe, get_client, propagate_attributes
from langfuse.openai import OpenAI

langfuse = get_client()
client  = OpenAI(base_url=os.getenv("OLLAMA_BASE_URL"), api_key="ollama")

# Simulated knowledge base
KNOWLEDGE_BASE = {
    "boundary value analysis": [
        "Test at exact boundaries and just outside them.",
        "For input 1-100: test 0, 1, 2, 99, 100, 101.",
    ],
    "equivalence partitioning": [
        "Divide inputs into classes that should behave identically.",
        "Test one value from each partition (valid and invalid).",
    ],
    "default": ["General QA knowledge base. No specific context found."],
}

SYSTEM_PROMPT = (
    "You are an experienced QA Architect with 20 years of software "
    "industry experience. Provide concise, practical answers with examples."
)


@observe(name="knowledge-retrieval", capture_output=False)
def retrieve_knowledge(query: str) -> list[str]:
    query_lower = query.lower()
    for key, docs in KNOWLEDGE_BASE.items():
        if key in query_lower:
            langfuse.update_current_span(
                output={"matched_key": key, "chunks": len(docs)},
                metadata={"source": "local-kb"},
            )
            return docs
    langfuse.update_current_span(output={"matched_key": "default", "chunks": 1})
    return KNOWLEDGE_BASE["default"]


@observe(name="llm-answer", as_type="generation", capture_input=False, capture_output=False)
def generate_qa_answer(question: str, context: list[str]) -> str:
    context_text = "\n".join(f"- {c}" for c in context)
    user_msg = f"Context:\n{context_text}\n\nQuestion: {question}"

    langfuse.update_current_generation(
        model=os.getenv("OLLAMA_MODEL"),
        model_parameters={"temperature": 0.3, "max_tokens": 500},
        input=[
            {"role": "system", "content": SYSTEM_PROMPT},
            {"role": "user",   "content": user_msg},
        ],
        metadata={"prompt_version": "2.0", "context_used": "true"},
    )

    response = client.chat.completions.create(
        model=os.getenv("OLLAMA_MODEL"),
        messages=[
            {"role": "system", "content": SYSTEM_PROMPT},
            {"role": "user",   "content": user_msg},
        ],
        temperature=0.3,
        max_tokens=500,
    )
    answer = response.choices[0].message.content

    langfuse.update_current_generation(
        output=answer,
        usage={
            "input":  response.usage.prompt_tokens,
            "output": response.usage.completion_tokens,
        },
    )
    return answer


@observe(name="qa-architect-pipeline")
def qa_architect_query(question: str) -> str:
    context = retrieve_knowledge(question)
    answer  = generate_qa_answer(question, context)
    return answer


if __name__ == "__main__":
    questions = [
        "Explain boundary value analysis with an example.",
        "What is equivalence partitioning?",
        "How do you prioritise test cases in an Agile sprint?",
    ]

    for q in questions:
        with propagate_attributes(
            trace_name="QA Architect Assistant",
            user_id="user-123",
            tags=["qa", "architect", "hour2-project"],
            metadata={
                "sdk_version": "4.7.1",
                "use_case": "qa-education",
            },
        ):
            result = qa_architect_query(q)
            print(f"\nQ: {q}\nA: {result[:200]}...\n")

    langfuse.flush()
    print("\n✓ All 3 traces sent to Langfuse. Check your dashboard!")
BASH run the project 
$ python 07_qa_architect_pipeline.py
What to look for in the dashboard: You should see 3 new traces, each named "QA Architect Assistant". Click any one — you'll see the nested tree: qa-architect-pipeline → knowledge-retrieval + llm-answer. The generation panel shows token counts, model parameters, and structured input/output. All three have the same user_id so they group in the Users view.
