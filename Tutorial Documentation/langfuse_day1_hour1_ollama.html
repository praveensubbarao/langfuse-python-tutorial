Create your Langfuse account
free tier · no credit card

Go to cloud.langfuse.com and sign up. The free tier gives you 50,000 observations/month — plenty for this entire 9-hour tutorial.

  1. Sign up at https://cloud.langfuse.com/auth/sign-up
  2. You land on a default project. Rename it langfuse-tutorial in Settings → Project.
  3. Go to Settings → API Keys and click Create new API key.
  4. Copy both the Secret Key (sk-lf-...) and Public Key (pk-lf-...). Store them — you only see the secret once.
Region matters. US users → https://us.cloud.langfuse.com  |  EU users → https://cloud.langfuse.com  |  Japan → https://jp.cloud.langfuse.com. The base URL goes in your .env in Step 4.
2
Install Ollama & pull a model
local LLM · no API key needed

Ollama is an open-source tool that runs LLMs locally. It exposes an OpenAI-compatible API at http://localhost:11434/v1 — which is exactly what Langfuse's drop-in wrapper will call.

Download Ollama from ollama.com/download for your OS (macOS, Linux, or Windows). Then pull a model. We recommend llama3.2 (2GB, fast) or mistral (4GB, very capable):

BASH terminal 
# After installing Ollama, pull a model (choose one):

$ ollama pull llama3.2      # ~2 GB — good default choice
# OR
$ ollama pull mistral        # ~4 GB — more capable

# Verify Ollama is running (keep this terminal open):
$ ollama serve               # starts the local server on :11434

Test that Ollama is responding before continuing. Open a second terminal and run:

BASH verify Ollama 
$ curl http://localhost:11434/v1/models

# Expected: JSON listing your pulled models
# {"object":"list","data":[{"id":"llama3.2",...}]}
Got JSON back? Ollama is live. You now have a local LLM serving on the same API format as OpenAI — this is the key insight that makes the Langfuse integration effortless.
3
Set up your Python environment
venv · pip install

Always work inside a virtual environment to keep your system Python clean.

BASH terminal 
# Create project folder
$ mkdir langfuse-tutorial && cd langfuse-tutorial

# Create a virtual environment
$ python -m venv .venv

# Activate it
$ source .venv/bin/activate        # Mac / Linux
# .venv\Scripts\activate             # Windows PowerShell

# Install required packages
$ pip install langfuse openai python-dotenv
Why install openai? Ollama exposes an OpenAI-compatible REST API. The openai Python package is the client for that API. Langfuse wraps the openai client, so no actual OpenAI account is needed — we just point the client at localhost:11434 instead.
SDK note: The Langfuse Python SDK was completely rewritten as v4 in March 2026. Make sure pip installs v4+ — run pip show langfuse and confirm the version is 4.x. It requires Python ≥ 3.10.

Also create a .gitignore now so you never accidentally commit secrets:

BASH terminal 
$ echo ".env" >> .gitignore
$ echo ".venv/" >> .gitignore
4
Configure your .env file
API keys · Langfuse + Ollama settings

Create a file named .env in your project folder and paste this in. Fill in your actual Langfuse keys from Step 1.

.ENV .env 
# ── Langfuse Cloud credentials ──────────────────────
LANGFUSE_PUBLIC_KEY="pk-lf-YOUR_PUBLIC_KEY_HERE"
LANGFUSE_SECRET_KEY="sk-lf-YOUR_SECRET_KEY_HERE"

# Your region base URL (pick the one matching your account):
LANGFUSE_BASE_URL="https://us.cloud.langfuse.com"   # 🇺🇸 US
# LANGFUSE_BASE_URL="https://cloud.langfuse.com"       # 🇪🇺 EU
# LANGFUSE_BASE_URL="https://jp.cloud.langfuse.com"    # 🇯🇵 Japan

# ── Ollama local server ──────────────────────────────
OLLAMA_BASE_URL="http://localhost:11434/v1"
OLLAMA_MODEL="llama3.2"                    # or "mistral", etc.
LANGFUSE_BASE_URL vs LANGFUSE_HOST: Use LANGFUSE_BASE_URL (not LANGFUSE_HOST) with the v4 SDK. The v4 rewrite standardised on BASE_URL. Using the wrong one is the most common setup mistake.
5
Send your first trace
01_first_trace.py · one import change

Create a file called 01_first_trace.py. The entire Langfuse integration is one import line different from plain OpenAI usage:

PYTHON 01_first_trace.py 
from dotenv import load_dotenv
import os
load_dotenv()   # reads your .env file into environment variables

# ✅ This ONE import is the entire Langfuse integration
# Langfuse wraps the OpenAI client — same API, auto-tracing
from langfuse.openai import OpenAI

# Point the client at your local Ollama server
client = OpenAI(
    base_url=os.getenv("OLLAMA_BASE_URL"),   # http://localhost:11434/v1
    api_key="ollama",                        # required by openai sdk, but ignored by Ollama
)

# Use the client exactly like you would use the OpenAI SDK
response = client.chat.completions.create(
    name="my-first-trace",              # readable label in Langfuse UI
    model=os.getenv("OLLAMA_MODEL"),    # e.g. "llama3.2"
    messages=[
        {"role": "system",  "content": "You are a helpful assistant."},
        {"role": "user",    "content": "What is Langfuse in one sentence?"},
    ],
    metadata={"tutorial_day": 1, "hour": 1},   # any custom data you want
)

print(response.choices[0].message.content)

Run it:

BASH terminal 
$ python 01_first_trace.py

# Expected output (your model's answer):
# Langfuse is an open-source LLM observability platform that helps...
Now check your Langfuse dashboard. Go to your project → Traces. Your trace should appear within a few seconds. Click it to explore. If nothing shows up after 30 seconds, check Step 4 — the LANGFUSE_BASE_URL is the most common culprit.

Concept: the Langfuse data model
traces · spans · generations

Before going further, understand what Langfuse actually captured. Every piece of data in Langfuse lives in a hierarchy:

Data model — what was captured
TRACE ← top-level container for one "user request" │ ├─ name "my-first-trace" ├─ metadata {tutorial_day: 1, hour: 1} ├─ timestamp 2026-06-05T... │ └─ GENERATION ← one LLM call inside the trace │ ├─ model "llama3.2" ├─ input [system msg, user msg] ├─ output "Langfuse is an open-source..." ├─ tokens {input: 34, output: 51} └─ latency 1240ms

In Hour 2 you'll add spans — which sit between traces and generations and represent non-LLM work like "retrieve from database" or "call external API". For now, think of it as:

Mental model
Trace = one user interaction ("answer this question") Span = any step inside it (retrieval, API call, etc.) Generation = specifically an LLM call (has model, tokens, cost)
Why does this matter? When your app gets complex (RAG pipelines, agents, multi-step workflows), every step gets its own span. You can see exactly which step was slow, which one failed, and how much each LLM call cost. This is the entire point of Langfuse.

8
Attach user IDs and tags
02_with_user.py · trace context

In real applications you always want to know which user triggered a trace. Create 02_with_user.py:

PYTHON 02_with_user.py 
from dotenv import load_dotenv
import os
load_dotenv()

from langfuse.openai import OpenAI
from langfuse import get_client

# Get the Langfuse client to create trace-level metadata
langfuse = get_client()

# Create a trace explicitly so we can attach user info
trace = langfuse.trace(
    name="user-question",
    user_id="user-123",             # ties this trace to a specific user
    tags=["tutorial", "day-1"],    # free-form tags for filtering
    metadata={"source": "cli"},
)

# Set up Ollama client pointing at local server
client = OpenAI(
    base_url=os.getenv("OLLAMA_BASE_URL"),
    api_key="ollama",
)

response = client.chat.completions.create(
    model=os.getenv("OLLAMA_MODEL"),
    messages=[
        {"role": "user", "content": "Explain observability in 2 sentences."},
    ],
    trace_id=trace.id,   # links this generation to our named trace
)

print(response.choices[0].message.content)

# Always flush at end of a script
# flush() ensures all buffered events are sent before the process exits
langfuse.flush()
BASH terminal 
$ python 02_with_user.py

In the Langfuse dashboard you'll now see Users in the left sidebar populate with user-123, and tags appear as filterable chips on your trace. This is the foundation of per-user analytics.
